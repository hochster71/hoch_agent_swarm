# HAF Existing Audit Capability Map v0.1

This document maps the existing audit and assurance capabilities in the `hoch_agent_swarm` repository, outlining their paths, capabilities, status vocabularies, weaknesses, and their migration disposition into the **HOCH Audit Factory (HAF) v0.1**.

## Capability Map Table

| Component Name | File Path | Capability | Evidence Produced | Current Status Vocabulary | Current Weaknesses | HAF Migration Disposition |
| :--- | :--- | :--- | :--- | :--- | :--- | :--- |
| **Hochster Runtime Audit** | `backend/hochster_runtime_audit.py` | Audits runtime tool calls, redactions, approval gates, and ledger events for validation presence and bypasses. | JSON audit report structure. | `"PASS"`, `"BLOCK"` | Lacks cryptographic integrity checks, schema validation, and POA&M integration. | **Wrap / Migrate**: Core logic is extracted and wrapped into executable HAF-TRUTH and HAF-QUEUE controls. |
| **HELM Audit Runner** | `scripts/helm_audit_runner.py` | Gathers evidence on re-execution, secrets, dependencies, egress, spend, and drift, and dispatches to Grok (LLM) for verification. | `audit_status.json` log and excerpt payload. | `"GREEN"`, `"FINDING"`, `"BLOCKED"`, `"RUNNING"`, `"PARTIAL"`, `"NEEDS_FOUNDER"` | Non-deterministic verification via LLM; does not use structured schemas, hashes, or circularity detection. | **Wrap / Migrate**: Replaces LLM-based Grok evaluation with deterministic HAF control checkers. |
| **Evidence Auditor Observer** | `backend/jspace/observers/evidence_auditor.py` | Monitors and audits events and logs in JSpace. | Observer event log entries. | Implicit (log warnings/infos) | Informational only; does not block unauthorized state transitions or enforce gate policies. | **Retain Unchanged**: Serves as passive background logging. |
| **Egress / Spend / Governance Pytests** | `tests/helm_runtime/` & `scripts/goal/` | Verifies and asserts specific gates like `test_founder_gate_requires_founder`, etc. | Pytest stdout/stderr and exit codes. | standard Pytest outcome (PASS/FAIL) | Separate, ad-hoc execution; results are not unified into a single certification registry or ConMon loop. | **Wrap**: Executed and verified by the HAF Assessment Runner under level-specific controls. |

## Conflicting Status Vocabularies
* Current systems mix `PASS`/`BLOCK`, `GREEN`/`FINDING`/`BLOCKED`/`PARTIAL`, and Pytest standard outputs.
* **HAF Solution**: Unify under the canonical certification registry vocabulary:
  * `PASS`
  * `PASS_WITH_LIMITATIONS`
  * `HOLD`
  * `FAIL`
  * `EXPIRED`
  * `NOT_ASSESSED`
  * `IN_PROGRESS`

## Identified Weaknesses & Gaps
1. **No Cryptographic Integrity**: Evidence files are referenced by path, but hashes are not verified/recomputed on load.
2. **Circular Evidence Paths**: A validator could technically consume evidence generated by itself without detection.
3. **No Continuous Monitoring (ConMon)**: Evidence lacks automated freshness validation linked to file/prompt/dependency mutations.
4. **No Formal Findings/POA&M Management**: Finding lifecycle is represented as raw text strings in status JSONs without tracking, milestones, or retest requirements.
